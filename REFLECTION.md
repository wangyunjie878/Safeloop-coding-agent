# 反思

本项目的目标是实现一个简化版 Coding Agent Harness：它不需要达到 Codex 或 Claude Code 的完整能力，但必须能在本地工作区内读写代码、执行命令、运行测试，并根据测试或工具反馈调整下一步行动。回头看，这个项目真正困难的地方并不是“调用一个大模型”，而是把一个看似自由的智能体行为拆成可验证、可限制、可复现的机制。最后形成的 SafeLoop 包含配置加载、事件日志、LLM 抽象、动作解析、护栏、工具分发、反馈分类、状态机、mock LLM、DeepSeek 接入、CLI、WebUI、Docker 和 CI/CD。这个过程让我更清楚地理解了 harness 项目为什么适合作为软件工程作业：它要求我把智能体能力具体化为代码边界，而不是只写提示词。

Superpowers 里对我帮助最大的是 brainstorming、writing-plans、test-driven-development 和 verification-before-completion。brainstorming 的价值主要体现在前期，它不断追问“到底要做什么”“目标用户是谁”“是不是必须像 Codex 一样完整”，迫使我把项目范围从模糊的“做一个 coding agent”缩小为“小型本地 CLI coding agent”。如果没有这一步，我很可能会一开始就追求大而全的 WebUI 或云端部署，导致核心机制反而不清晰。writing-plans 则把设计拆成可执行任务，让每个 task 都有文件路径、预期实现和验证命令。它看起来很形式化，但在后期确实降低了偏航风险：当我担心 WebUI 太简陋、CLI 不像 opencode、DeepSeek 超时或问答会误写文件时，都能把问题归入新的小 task，而不是推翻整个架构。

我认为这些流程既有形式大于实质的部分，也有真正提升质量的部分。形式化的一面是，PLAN.md 中有些早期 checkbox 后来没有完全同步勾选，实际证据反而记录在 commit、AGENT_LOG 和后续 task 中。这说明“写很多流程文档”本身不等于高质量，甚至可能产生维护负担。但实质有效的一面也很明显：SPEC.md 和 PLAN.md 让冷启动 agent 能只凭文档完成 Task 1，同时暴露出 Makefile 在 Windows 上不可用、build backend 写得不够明确、占位命令 exit code 有歧义等问题。这些反馈如果只靠我和主 agent 的共享上下文，很可能不会发现。因此，规约和计划的价值不在于漂亮，而在于它们能否让一个陌生执行者少猜测。

TDD 在这个项目里更像是安全带，而不是负担。因为 harness 的很多错误不是语法错误，而是行为边界错误：危险命令是否真的被护栏拦截，文件路径是否会逃出 workspace，测试失败是否会进入下一轮反馈，DeepSeek 的响应是否必须是一个 JSON 工具动作，凭据是否会泄漏到日志。先写失败测试让我能看到“当前系统确实没有这个能力”，再写最小实现。例如机制演示中，mock LLM 先尝试执行危险命令，系统必须产生 `guardrail_blocked`；随后运行测试失败，反馈进入下一次 LLMRequest；下一步再 patch 文件并重新运行测试。这个过程如果只手动试，很容易只验证“看起来能跑”，而不是验证核心机制真的存在。TDD 的缺点是节奏会慢，尤其在文档类或 CLI 文案类修改时显得繁琐，但对安全边界和反馈循环这种关键机制来说，它明显是放大器。

subagent-driven development 的体验比较复杂。它的优点是能把 task 粒度逼小：一个 subagent 只负责模型与配置，另一个负责文件工具，另一个负责命令工具，这样上下文更干净，PR 证据也更符合课程要求。它也让我意识到 task 颗粒度必须足够具体，最好能在一次会话内完成，并且包含明确失败测试和验证命令。否则 subagent 很容易根据自己的理解补设计，或者在超时后留下半成品。项目中也出现过 subagent 超时、PR 创建权限不足、review agent 用量限制等问题，最后仍需要 controller 手动核对 diff、测试和文档。因此我不会把 subagent 理解成“自动完成开发”，它更像一个可替换的执行者，必须被 SPEC、PLAN、测试和 review 约束。

最有效的 prompt/context 策略是给 agent 明确的边界，而不是只给目标。比如“不要做完整 Codex，只做能读写代码、执行命令、运行测试、根据测试结果自我修正的小型 agent”；“不要把真实 key 写进仓库”；“mock LLM 测试必须离线确定性”；“用户 cd 到哪个目录，agent 就操作哪个目录”；“纯解释问题不要写文件，直接中文回答”。这些限制比“做得好一点”“像 opencode 一样”更有效。后期 DeepSeek chat 能用以后，我发现提示词仍然会影响真实模型行为：如果只告诉它必须返回工具 JSON，它会把解释类问题也变成写文件任务；只有明确加入 information-only 请求应该 `finish` 且不改文件，行为才更符合用户预期。这也说明 harness 不能完全摆脱 prompt，但关键安全机制不能只依赖 prompt。

凭据与分发要求让我更早考虑真实使用场景。DeepSeek key 不能提交进仓库，也不应该写进配置样例，所以项目使用 keyring、环境变量和 dotenv 作为不同环境下的凭据来源，并在日志和事件中做脱敏。Docker 和 GitHub Actions 也改变了我对“完成”的理解：本地能运行不代表别人能运行，真实 LLM 能跑不代表 CI 能跑。最终 CI 使用 mock LLM，运行 `python -m pytest`、`python -m safeloop demo` 和 Docker build，这样老师可以在 Actions 里看到可复现证据，而不需要我的电脑在线或提供真实 key。这里我学到的一个重要点是，离线确定性测试不是退而求其次，而是 agent harness 的核心交付方式。

如果重新做一次，我会在三方面改进。第一，我会更早把最终产品形态定为 CLI-first，而不是先做一个很弱的 WebUI。WebUI 虽然满足接口展示，但用户真正期待的是像 opencode 一样在当前目录对话和改代码。第二，我会更早统一任务清单和最终验收清单，避免后期 PLAN.md 中出现一些早期步骤未勾选但实际已经由后续 commit 覆盖的情况。第三，我会把“自然语言回答”和“代码修改任务”的分流作为一开始的核心机制，而不是等真实使用时发现它只会写文件再修正。

总体来说，Superpowers 方法论对我的最大帮助是把 AI 协作从“让 agent 自己发挥”变成“用规约、计划、测试、日志、PR 和 CI 管住 agent”。我对它的批判是，它有时假设流程越完整越好，但学生项目里时间和认知成本有限，过重的流程会让人误以为文档数量就是质量。真正有效的不是机械执行每个仪式，而是保留那些能产生证据的环节：冷启动验证暴露隐性假设，TDD 证明机制存在，AGENT_LOG 记录人工干预，CI 证明交付可复现。这个项目最后能跑起来，靠的不是某个大模型一次性写对，而是把许多不确定的智能体行为压进了可测试的软件工程循环里。
