# AI4SE 期末项目 · B · 非 harness 应用类项目

> 本文件须与《AI4SE 期末项目 · 通用要求》拼接阅读。适用于选择"非 harness 主题"的学生：Web 应用、移动应用、微信小程序、桌面工具、AI 原生应用、数据平台、面向开发者的工具等。
>
> 你的完整要求 = 通用要求 + 本文件。本方向不构建 agent harness，因此通用要求中"harness 特定"的标注（领域与机制设计、harness 内核、mock-LLM 测试、机制演示）对你**不适用**；其余通用要求（凭据安全、分发、Superpowers 全流程、五份过程文档、冷启动验证、CI、反思）**全部照常适用**。

---

## B.1 项目要求

构建一个真实、有规模、解决真实问题的软件项目。

- 必须能在 30 秒内向陌生人说清"这东西为什么有人会用"。
- 至少 3 个以上职责清晰的功能模块、可一键运行的测试。
- 不接受纯 demo 或玩具级项目。
- 评价重心是工程深度与完成度，而非代码量（见通用要求 §3.4）。

## B.2 若项目含 agent / 智能体部分

判断你的项目里有没有"agent"：**agent 指能自主多轮决策、自主调用工具去完成任务的部分**（有循环、有工具自主调用、有据反馈自我修正）。

- **若有 agent 部分**：那部分须参照 harness 的实现边界——agent 的主循环、工具分发、治理护栏要由你自己编码，且应满足"移除真实 LLM 后仍能用确定性单元测试验证"的判定标准（详见 `AI4SE_Final_Project_A_Coding_Agent_Harness.md` 的 §A.4）。不能仅在现成 agent 框架上配置提示词来充当这部分功能。
- **若无 agent 部分**（例如只是单轮调用 LLM 做生成 / 问答 / 摘要，没有自主循环与工具自主调用）：则不构成 agent，按普通功能模块对待，无上述约束。

## B.3 凭据与分发

通用要求 §3.1（凭据安全存储）与 §3.2（分发）同样适用：

- 若调用 LLM 或任何需鉴权的付费 API，key 必须安全存储，绝不硬编码或提交进仓库。
- 带服务端的项目适合走容器分发（`docker build` + `docker run`）并可选云部署；纯客户端 / 工具类项目可走二进制或包管理器分发。

## B.4 前端

若项目包含任何前端 / UI（Web 页面、移动端界面、营销页、文档站、控制台等），强烈推荐使用 **[Open Design](https://github.com/nexu-io/open-design)** 进行界面开发，并在 SPEC 中说明所选设计系统与 skill（详见通用要求 §3.6）。纯后端 / 纯 CLI 项目可豁免。

## B.5 交付物

与通用要求 §五的清单一致，但**不含** harness 内核、mock-LLM 单元测试、机制演示这三项。其余（SPEC / PLAN / SPEC_PROCESS / 源码与 PR 历史 / 分发产物 / README / AGENT_LOG / CI / REFLECTION）照常提交。核心功能须有可一键运行的测试与 CI。
